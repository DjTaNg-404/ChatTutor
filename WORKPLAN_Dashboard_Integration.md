# ChatTutor Dashboard 对接排期表（MVP）

> 目标：在最短周期内把 `Design_Web_Dashboard` 从 UI 原型升级为可用的学习 Dashboard，完成“对话可用、上下文按任务可追踪、笔记可沉淀”三条主线。

## 1. 范围与优先级

### P0（必须完成）
- 对话接口联调（前端真实调用 `/api/v1/chat`）
- 任务维度会话隔离（按 `task_id` 组织会话）
- 会话历史查询（按任务、按日期）
- 每日笔记与任务总笔记基础读写

### P1（建议完成）
- 总结质量优化（Prompt + 输出结构）
- 接口错误码与前端异常提示统一
- 基础回归测试与验收清单固化

### P2（后续迭代）
- 独立 Note Agent（当结构化笔记需求变复杂时）
- 数据库替换 JSON 文件存储
- 多用户与权限体系

## 3. 分阶段任务清单（可直接执行）

## 阶段A：契约设计（Day 1）

### 任务
- 定义会话标识：`session_id = {user_id}:{task_id}:{yyyyMMdd}`（或序号）
- 扩展 `/api/v1/chat` 入参：增加 `task_id`（可选 `user_id`）
- 约定标准响应：`reply/is_concluded/session_id/task_id`

### 验收
- 团队对命名与字段达成一致
- 后端与前端都按同一字段开发

---

## 阶段B：对话联调（Day 2-3）

### 任务
- 在 `TutorSession` 使用真实 `fetch`/`axios` 调用 `/api/v1/chat`
- 处理加载态、失败重试、空消息校验
- 对接 `is_concluded` 触发“结束会话”提示

### 验收
- 用户可在 Dashboard 连续发送 10+ 轮对话
- 刷新页面后继续同一任务对话不丢上下文

---

## 阶段C：任务化上下文与历史（Day 4-5）

### 任务
- 新增历史接口（建议）：
  - `GET /api/v1/tasks/{task_id}/sessions`
  - `GET /api/v1/sessions/{session_id}/messages`
- 将侧边栏任务、时间线、历史页改成真实数据

### 验收
- 同一用户多任务互不串线
- 任务页面切换后可看到对应历史

---

## 阶段D：笔记能力（Day 6-7）

### 任务
- 新增笔记接口（建议）：
  - `GET /api/v1/notes/daily?task_id=&date=`
  - `PUT /api/v1/notes/daily`
  - `GET /api/v1/notes/task?task_id=`
  - `PUT /api/v1/notes/task`
- 结束会话后将 `summary_output` 归档到每日笔记

### 验收
- 每日笔记页能查看/编辑/保存
- 任务笔记页可汇总历史关键点

---

## 阶段E：总结能力优化（Day 8）

### 任务
- 先迭代现有 `summary_node` Prompt（不立即拆新 Agent）
- 输出结构固定：核心洞察、概念解析、下一步行动
- 增加“总结质量检查”规则（长度、信息密度、重复率）

### 验收
- 3 组真实会话的总结可读性明显提升
- 页面展示无需大量手工修正

---

## 阶段F-G：回归与发布（Day 9-10）

### 任务
- 走通 E2E 主流程：新任务 → 对话 → 结束 → 笔记 → 历史回看
- 修复高优问题（P0/P1）
- 补齐 README 运行与联调说明

### 验收
- 主流程 100% 可复现
- 已知问题有记录与规避方案

## 5. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 会话ID设计不稳 | 历史串线、上下文错乱 | 先冻结命名规则再开发 |
| 前后端字段不一致 | 页面报错、联调反复 | Day1 输出接口文档并锁版本 |
| 总结质量波动 | 笔记可读性差 | 先调 Prompt，再考虑独立 Agent |
| JSON 存储膨胀 | 查询变慢 | 预留 DB 迁移方案（P2） |
