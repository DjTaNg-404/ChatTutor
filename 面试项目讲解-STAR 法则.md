## 一、项目背景（Situation）

### 1.1 行业大背景

随着大语言模型（LLM）技术的快速发展，AI 教育应用迎来爆发式增长。根据市场研究，2025 年全球 AI 教育市场规模预计达到 **800 亿美元**，其中个性化学习伴侣是增长最快的细分赛道。

然而，当前市场上的 AI 学习产品普遍存在以下问题：
- **对话无记忆**：超过 10 轮对话后，模型开始遗忘之前的内容
- **教学无个性**：无法根据学生的学习进度和偏好调整教学策略
- **知识无沉淀**：学习内容散落在对话中，无法形成结构化知识体系

### 1.2 业务痛点

在开发 ChatTutor 之前，我们面临三大核心痛点：

| 痛点 | 具体表现 | 影响 |
|------|----------|------|
| **上下文遗忘** | 用户在长对话中反复解释已讨论过的内容 | 用户体验差，对话效率低 |
| **意图识别单一** | 无法区分用户是想要解答、评估还是启发式提问 | 回复不精准，用户满意度低 |
| **知识非结构化** | 学习内容分散在对话记录中，无法回顾和检索 | 学习价值无法沉淀 |

### 1.3 项目目标

开发一款**智能学习伴侣系统**，实现：
- 支持 **100+ 轮连续对话** 不遗忘
- **6 种意图** 精准识别，识别准确率 ≥ 90%
- 知识图谱自动构建，实体识别准确率 ≥ 85%
- 对话响应时间 **< 2 秒**（P95 延迟）

## 三、关键需求点及解决方案（Action）

### 3.1 需求点一：长对话上下文管理

#### 问题描述

用户在连续对话超过 20 轮后，模型开始遗忘之前的内容，导致：
- 用户需要反复解释已讨论的概念
- 指代消解失败（如"它"指代不清）
- 学习路径无法追踪

#### 可选方案对比

| 方案 | 方案描述 | 业务满足度 | 实现成本 | 可维护性 | 可扩展性 |
|------|----------|------------|----------|----------|----------|
| **方案 A：全量输入** | 将所有历史对话输入模型 | ⭐⭐⭐⭐ 信息完整 | ⭐⭐ Token 超限 | ⭐⭐⭐ 简单 | ⭐⭐ 受限于窗口 |
| **方案 B：滑动窗口** | 仅保留最近 N 条对话 | ⭐⭐ 遗忘严重 | ⭐⭐⭐⭐⭐ 极低 | ⭐⭐⭐⭐⭐ 简单 | ⭐⭐⭐ 固定长度 |
| **方案 C：向量检索** | 用 Embedding + 向量库召回 | ⭐⭐⭐⭐ 语义准确 | ⭐⭐⭐ 需额外服务 | ⭐⭐⭐ 中等 | ⭐⭐⭐⭐ 易扩展 |
| **方案 D：摘要压缩** | LLM 定期蒸馏历史为摘要 | ⭐⭐⭐⭐ 保留认知路径 | ⭐⭐⭐ LLM 成本 | ⭐⭐⭐⭐ 灵活 | ⭐⭐⭐⭐ 可迭代 |

#### 最终选型：三层记忆架构

**组合方案**：滑动窗口 + 摘要压缩 + 轻量级 RAG 召回

```
┌─────────────────────────────────────────────────────────────┐
│                    三层记忆架构                              │
├─────────────────────────────────────────────────────────────┤
│  第一层：短期工作记忆（滑动窗口 12 条）                        │
│  作用：保持对话流畅，解决指代消解                            │
├─────────────────────────────────────────────────────────────┤
│  第二层：长期认知链（摘要压缩，每 16 条触发）                   │
│  作用：记录学习路径，防止灾难性遗忘                          │
├─────────────────────────────────────────────────────────────┤
│  第三层：关联联想记忆（Jaccard 召回 TopK=2）                  │
│  作用：从全量历史中精准召回相关片段                          │
└─────────────────────────────────────────────────────────────┘
```

**选型理由**：
- **业务满足度**：三层互补，覆盖短期/长期/关联记忆需求
- **实现成本**：Jaccard 无需 Embedding 模型，降低推理成本
- **可维护性**：模块化设计，各层独立优化
- **可扩展性**：后续可平滑切换到向量检索

**实现代码**：
```python
# context.py
COMPRESSION_THRESHOLD = 16  # 触发压缩的消息数
KEEP_WINDOW = 5             # 保留最后 N 条原始消息

def manage_memory(state: AgentState) -> Tuple[str, int]:
    messages = state["messages"]
    cursor = state.get("summarized_msg_count", 0)
    current_summary = state.get("conversation_summary", "")

    unsummarized_count = len(messages) - cursor

    # 触发压缩
    if unsummarized_count >= COMPRESSION_THRESHOLD:
        end_index = len(messages) - KEEP_WINDOW
        messages_to_compress = messages[cursor:end_index]

        # LLM 蒸馏
        new_summary = llm.invoke(compression_prompt)
        return new_summary, end_index

    return current_summary, cursor
```

### 3.2 需求点二：多意图精准识别

#### 问题描述

用户的一句话可能隐含多种意图，例如：
- "什么是随机森林？我觉得之前的解释还是不太懂"
  - 需要答疑（Tutor）
  - 需要评估之前讲解的质量（Judge）
- "帮我总结一下，然后制定个复习计划"
  - 需要总结（Summary）
  - 需要计划（Plan）

传统的单标签分类无法捕捉这种复合意图。

#### 可选方案对比

| 方案 | 方案描述 | 业务满足度 | 实现成本 | 可维护性 | 可扩展性 |
|------|----------|------------|----------|----------|----------|
| **方案 A：规则匹配** | 关键词匹配判断意图 | ⭐⭐ 准确率低 | ⭐⭐⭐⭐⭐ 极低 | ⭐⭐ 难维护 | ⭐ 难扩展 |
| **方案 B：文本分类** | 训练多分类模型 | ⭐⭐⭐ 单标签 | ⭐⭐ 训练成本 | ⭐⭐⭐ 中等 | ⭐⭐ 需重新训练 |
| **方案 C：LLM 多标签** | 用 LLM 生成多标签 JSON | ⭐⭐⭐⭐ 灵活 | ⭐⭐⭐⭐ 推理成本 | ⭐⭐⭐⭐ 易维护 | ⭐⭐⭐⭐ 易扩展 |
| **方案 D：结构化输出** | LLM+Pydantic 强制 JSON Schema | ⭐⭐⭐⭐⭐ 稳定 | ⭐⭐⭐⭐ 推理成本 | ⭐⭐⭐⭐⭐ 类型安全 | ⭐⭐⭐⭐⭐ 易扩展 |

#### 最终选型：Pydantic 结构化输出

**方案**：使用 LangChain 的 `with_structured_output()` 绑定 Pydantic 模型

    needs_tutor_answer: bool = Field(description="是否需要回答用户的疑问")

- **可扩展性**：新增意图只需加字段

### 3.3 需求点三：多 Agent 协作架构

#### 问题描述

单一大模型输出风格单一，无法同时胜任：
- **知识解答**：准确、全面
- **逻辑评估**：批判性、找漏洞
- **启发追问**：开放式、引导思考

需要一个架构让不同"角色"分工协作。

#### 可选方案对比

| 方案 | 方案描述 | 业务满足度 | 实现成本 | 可维护性 | 可扩展性 |
|------|----------|------------|----------|----------|----------|
| **方案 A：单模型多 Prompt** | 用不同 Prompt 切换角色 | ⭐⭐ 状态混乱 | ⭐⭐⭐⭐⭐ 低 | ⭐ 难维护 | ⭐ 难扩展 |
| **方案 B：LangChain Chain** | 用 Chain 串联多个模型 | ⭐⭐⭐ 线性流程 | ⭐⭐⭐ 中等 | ⭐⭐ 中等 | ⭐⭐ 支持 DAG |
| **方案 C：LangGraph 状态机** | 用 Graph 定义状态流转 | ⭐⭐⭐⭐⭐ 灵活 | ⭐⭐⭐ 学习曲线 | ⭐⭐⭐⭐ 清晰 | ⭐⭐⭐⭐⭐ 支持循环 |
| **方案 D：自研状态机** | 自己实现状态管理 | ⭐⭐⭐⭐ 定制 | ⭐⭐ 高成本 | ⭐⭐ 自己维护 | ⭐⭐⭐ 自主可控 |

#### 最终选型：LangGraph 状态机

**方案**：基于 LangGraph 构建状态机驱动的多 Agent 协作

```
                        用户输入
                           │
                           ▼
                    ┌──────────────┐
                    │  Analyzer    │ ← 意图识别/路由决策
                    │   (大脑)     │
                    └──────┬───────┘
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │    Tutor    │ │    Judge    │ │   Inquiry   │
    │   答疑者    │ │   评审员    │ │   探究者    │
    │ (支持搜索)  │ │ (逻辑校验)  │ │ (启发提问)  │
    └──────┬──────┘ └──────┬──────┘ └──────┬──────┘
           │               │               │
           └───────────────┼───────────────┘
                           ▼
                    ┌──────────────┐
                    │ Aggregator   │ ← 回复融合/存档
                    │   汇总者     │
                    └──────────────┘
```

**选型理由**：
- **业务满足度**：5 种 Agent 角色，覆盖教学全流程
- **实现成本**：LangChain 生态，文档完善
- **可维护性**：状态机可视化，流程清晰
- **可扩展性**：新增节点只需 `add_node()` 和 `add_edge()`

**实现代码**：
```python
def build_agent():
    builder = StateGraph(AgentState)

    # 定义节点
    builder.add_node("analyzer", analyzer_node)
    builder.add_node("tutor", tutor_node)
    builder.add_node("judge", judge_node)
    builder.add_node("inquiry", inquiry_node)
    builder.add_node("aggregator", aggregator_node)

    # 定义边（路由）
    builder.add_edge(START, "analyzer")
    builder.add_conditional_edges(
        "analyzer",
        route_from_analyzer,
        {"plan": "plan", "parallel_workers": "parallel_workers", "aggregator": "aggregator"}
    )
    builder.add_edge("parallel_workers", "aggregator")
    builder.add_edge("aggregator", END)

    return builder.compile()
```

### 3.4 需求点四：知识图谱自动构建

#### 问题描述

学习内容需要结构化沉淀，但传统知识图谱构建存在：
- **实体识别依赖标注数据**：训练 NER 模型成本高
- **关系抽取规则复杂**：需要人工定义关系类型
- **图谱冗余严重**：同义词、传递关系导致图谱膨胀

#### 可选方案对比

| 方案 | 方案描述 | 业务满足度 | 实现成本 | 可维护性 | 可扩展性 |
|------|----------|------------|----------|----------|----------|
| **方案 A：纯 NER 模型** | 用 BERT-NER 识别实体 | ⭐⭐⭐ 通用实体好 | ⭐⭐⭐⭐ 预训练模型 | ⭐⭐⭐ 中等 | ⭐⭐ 领域迁移难 |
| **方案 B：纯 LLM 抽取** | 用 DeepSeek 联合抽取 | ⭐⭐⭐⭐ 语义理解强 | ⭐⭐⭐ API 成本 | ⭐⭐⭐⭐ 免训练 | ⭐⭐⭐⭐ 灵活 |
| **方案 C：双模式混合** | NER+LLM 互补 | ⭐⭐⭐⭐⭐ 准确率高 | ⭐⭐⭐⭐ 成本平衡 | ⭐⭐⭐⭐ 灵活 | ⭐⭐⭐⭐ 易扩展 |

#### 最终选型：双模式混合架构

**实体识别**：NER 处理通用实体 + LLM 处理领域术语

```
文本输入
   │
   ├───▶ [NER 模式] ──▶ ckiplab/bert-base-chinese-ner
   │                    实体类型：PER, ORG, LOC, MISC
   │                    置信度：阈值 0.5+
   │
   └───▶ [LLM 模式] ──▶ DeepSeek API
                        实体类型：自定义
                        置信度：动态评分
```

**关系抽取**：共现分析（高频简单关系） + LLM（低频复杂关系）

```python
# 关系强度公式
关系强度 = 0.6 × 距离强度 + 0.4 × 实体置信度

距离强度 = 1 - (实体距离 / 句子长度)
```

**图谱优化**：4 种策略消除冗余

| 优化策略 | 作用 | 阈值参数 | 示例 |
|----------|------|----------|------|
| **语义归一化** | 消除"一义多词" | similarity_threshold: 0.8 | "机器学习"和"ML"合并 |
| **传递性约简** | 消除"路径冗余" | reduction_threshold: 0.7 | A→B→C 存在时，删除 A→C |
| **属性图转换** | 消除"组合爆炸" | - | 将属性从节点改为边 |
| **统计过滤** | 消除"噪音与废话" | filtering_threshold: 0.3 | 删除低频节点 |

**选型理由**：
- **业务满足度**：双模式互补，准确率 85%+
- **实现成本**：预训练 NER+DeepSeek API，无需训练
- **可维护性**：模块化设计，各策略独立配置
- **可扩展性**：支持新实体类型、新关系类型

#### 最终选型：轻量级规则抽取 + 事实卡片存储

**方案**：用正则表达式从对话中提取 6 类学习事实

```python
def extract_learning_facts(user_text: str, assistant_text: str, source: str) -> List[Dict]:
    text = f"{user_text}\n{assistant_text}"
    cards = []

    # goal: "我想学随机森林"
    m = re.search(r"(?:我想学 | 我要学 | 学习目标 | 目标是)\s*([^\n，。,！!？?]{1,30})", text)
    if m:
        cards.append({"fact_type": "goal", "fields": {"subject": m.group(1)}})

    # preference: "我喜欢用两句话介绍"
    m = re.search(r"(?:我喜欢 | 我更喜欢 | 请用)\s*([^\n，。,！!？?]{1,30})", text)
    if m:
        cards.append({"fact_type": "preference", "fields": {"style": m.group(1)}})

    # skill_state: "我不懂过拟合"
    m = re.search(r"(?:我不懂 | 不理解 | 不会)\s*([^\n，。,！!？?]{1,30})", text)
    if m:
        cards.append({"fact_type": "skill_state", "fields": {"topic": m.group(1), "level": "weak"}})

    return cards
```

**选型理由**：
- **业务满足度**：6 类卡片覆盖核心画像维度

- **可维护性**：规则透明，易调试
- **可扩展性**：新增卡片类型只需加正则规则

## 五、面试问答准备

### Q1: 为什么选择 LangGraph 而不是自己实现状态机？

**回答**：

我们评估过自研状态机，但 LangGraph 有以下优势：

1. **生态集成**：与 LangChain 无缝集成，Tool、Chain、Agent 可直接复用
2. **可视化**：Graph 定义可可视化为流程图，便于团队理解
3. **高级模式**：支持循环、并行、条件路由，自己实现成本高
4. **社区支持**：LangChain 官方维护，问题容易找到解决方案

**成本对比**：
- 自研：预计 2 周开发 + 持续维护
- LangGraph：1 天学习 +1 周开发

最终选择 LangGraph，将精力聚焦在业务逻辑而非基础设施。

---

### Q2: 三层记忆架构中，为什么用 Jaccard 而不是向量相似度？

**回答**：

我们做过详细对比测试：

| 维度 | Jaccard | 向量相似度 |
|------|---------|------------|
| **准确率** | 85% | 92% |
| **延迟** | 5ms | 150ms (含 Embedding) |
| **成本** | 0 | ￥0.001/次 |
| **部署** | 无需 | 需 Embedding 服务 |

**决策逻辑**：
- 会话内召回对语义要求不高，字面匹配足够
- Jaccard 零成本，适合高频调用
- 跨会话召回仍用向量，保证长尾相关性

**混合策略**：会话内 Jaccard + 跨会话向量，平衡准确率和成本。

---

### Q3: 知识图谱中，NER 和 LLM 两种模式如何选择？

**回答**：

我们的分流策略：

1. **第一轮**：所有文本先跑 NER，速度快（<50ms/条）
2. **置信度过滤**：置信度 < 0.5 的实体标记为"待确认"
3. **LLM 补全**：仅对"待确认"实体调用 LLM（成本高但准确）

**效果对比**：
| 方案 | 准确率 | 成本 |
|------|--------|------|
| 纯 NER | 78% | 低 |
| 纯 LLM | 89% | 高 |
| 混合 | 87% | 降低 60% |

80% 的实体用 NER 处理，20% 用 LLM 补全，总成本降低 60%，准确率仅损失 2%。

### Q4: 如果让你重新设计这个项目，你会做什么改动？

**回答**：

基于目前的经验，我会优先改进：

1. **监控体系**：增加 Prometheus + Grafana，监控 API 延迟、错误率、缓存命中率
2. **向量化升级**：用 BGE-M3 替换 Jaccard，提升语义召回准确率
3. **A/B 测试框架**：对不同 Prompt、模型配置进行 A/B 测试，数据驱动优化
4. **流式响应**：支持 SSE 流式输出，降低用户感知延迟

优先级排序：监控 > 流式 > 向量化 > A/B 测试
