## 二、面试逐字稿（5-8 分钟版）

---

各位面试官好，我这段实习是在一家做 **AI 搜题软件** 的教育科技公司，产品主要面向大学生，拍照搜题出解析。

我进去的时候，团队面临三个核心问题：

### 问题一：微调数据质量差

当时他们想做模型微调，但原始数据很乱——一道题多个版本重复、格式不统一、难度没标注。

**我的工作**：
1. **多题去重**：用 Gemini 做语义相似度判断，把同一道题的不同版本聚成一类
2. **质量筛选**：设计规则过滤掉解析不完整、公式乱码的题目
3. **通道划分**：按学科（高数/线代/概率论）和难度（简单/中等/困难）打标签

**成果**：清洗出 10 万 + 高质量题目，微调后模型在垂直领域的准确率提升了 15%。

---

### 问题二：用户反馈处理效率低

每天有几百条用户报错，人工处理很慢，也不知道哪些是高频问题。

**我的工作**：
- 用 Gemini 自动化归类用户反馈，把相似的错误归成一类
- 每周输出测评报告，统计 top10 高频错误场景
- 推动产品修复了 8 类高频问题，比如"公式识别错误""多解情况漏判"

**成果**：用户满意度从 75% 提升到 87%，周度报告成了产品迭代的固定输入。

---

### 问题三：模型选型没有数据支撑

当时公司只用 Gemini，但市面上新模型不断出来，要不要换、换哪个，没有依据。

**我的工作**：
设计了一个 **多模型评测框架**，这是技术含量最高的部分。

**架构设计**：

```
│                    评测框架架构                              │
├─────────────────────────────────────────────────────────────┤

│  │   Gemini    │  │   GPT-4     │  │   Claude    │         │
│  │   DeepSeek  │  │   Kimi      │  │  通义千问   │         │

│         └────────────────┼────────────────┘                  │
│                          ▼                                   │
│              ┌───────────────────────┐                      │

```

**技术实现**：
- 用 **LangChain 的 ChatModel 抽象** 统一不同模型的调用接口
- **FastAPI** 提供评测接口，传入题目和候选模型列表
- **配置文件** 管理各模型的 API Key 和参数，加新模型只要改配置

**核心代码**：
```python
# config.yaml
models:
  gemini:
    provider: gemini
    api_key: ${GEMINI_KEY}
    model: gemini-1.5-pro

  deepseek:
    provider: deepseek
    api_key: ${DEEPSEEK_KEY}
    model: deepseek-chat

  gpt4:
    provider: openai
    api_key: ${OPENAI_KEY}
    model: gpt-4-turbo

# evaluator.py
from langchain.chat_models import ChatGemini, ChatDeepSeek, ChatOpenAI

MODEL_REGISTRY = {
    "gemini": ChatGemini,
    "deepseek": ChatDeepSeek,
    "gpt4": ChatOpenAI,
}

def get_model(model_name: str):
    config = load_config()[model_name]
    return MODEL_REGISTRY[config["provider"]](**config)
```

**评测维度**：

| **准确率** | 答案正确率 | 40% |
| **解析质量** | 步骤完整性、可读性 | 30% |
| **延迟** | P95 响应时间 | 15% |
| **成本** | 每千题 API 费用 | 15% |

**成果**：
- 单次评测效率提升 5 倍（从 1 天到 2 小时）
- 输出 6 份横向对比报告
- 推动切换至 DeepSeek，API 成本降低 40%

### 三、遇到的困难与解决

**困难一：数据格式混乱**

原始数据有 JSON、CSV、甚至 Excel，公式有的用 LaTeX 有的用图片。

**解决**：
- 写脚本统一转成标准 JSON 格式
- 用 Gemini 把图片公式转成 LaTeX
- 设计 schema 校验保证输出一致

**收获**：数据清洗是脏活累活，但直接决定模型效果上限。

---

**困难二：评测结果波动大**

同一个模型，不同时间跑结果不一样，没法比。

**解决**：
- 固定 temperature=0 保证输出稳定
- 每个场景跑 5 次取平均
- 建了 500 题的标准测试集，每次全量跑

**收获**：评测的可靠性比单次结果更重要。

---

**困难三：LangChain 封装有坑**

有些模型的 API 和 LangChain 抽象对不上，比如 Gemini 的多模态输入。

**解决**：
- 自己继承 ChatModel 基类写适配器
- 特殊能力用自定义参数透传

**收获**：框架是工具，理解底层原理才能灵活定制。

### 五、简历问答准备

**Q1: 你说的多题去重具体怎么做？**

A: 用 Gemini 做语义嵌入，计算题目之间的余弦相似度。具体是：
1. 把题目文本输入 Gemini 获取 embedding
2. 两两计算余弦相似度
3. 相似度>0.9 的聚成一类
4. 每类选一道解析最完整的作为标准题

用的是 Gemini 的 `embed_content` 接口，批量处理效率很高。

---

**Q2: 评测框架为什么用 LangChain？**

A: 主要是看中它的抽象能力。LangChain 的 ChatModel 基类定义了统一的 `invoke()` 接口，我只要针对不同模型写适配器就行。

不用 LangChain 也可以，但每次加新模型都要改调用逻辑，用 LangChain 之后加模型只改配置文件。

---

**Q3: 你们为什么选 DeepSeek？**

A: 从四个维度评测：
- **准确率**：DeepSeek 和 Gemini 持平，略低于 GPT-4
- **延迟**：DeepSeek 平均 500ms，Gemini 是 800ms
- **成本**：DeepSeek 是 Gemini 的 1/3
- **中文能力**：DeepSeek 明显优于 Gemini

综合性价比，DeepSeek 是最优选择。
