### 一、Agent 架构相关（6 题）

---

#### Q1: 为什么选择 LangGraph 而不是自研状态机或 LangChain Chain？

**考察点**：技术选型能力、架构理解

**回答**：
我们评估过三种方案：

| 方案 | 开发成本 | 可维护性 | 可扩展性 |
|------|----------|----------|----------|
| 自研状态机 | 2 周 + | ⭐⭐ | ⭐⭐ |
| LangChain Chain | 1 周 | ⭐⭐⭐ | ⭐⭐（线性流程） |
| **LangGraph** | **1 周** | **⭐⭐⭐⭐** | **⭐⭐⭐⭐⭐（支持循环）** |

**选择 LangGraph 的核心理由**：
1. **生态集成**：与 LangChain 无缝集成，Tool、Chain 可直接复用
2. **可视化**：Graph 定义可可视化为流程图，便于团队理解
3. **高级模式**：支持循环、并行、条件路由——我们的多 Agent 协作需要循环追问场景
4. **社区支持**：LangChain 官方维护，问题容易找到解决方案

**成本对比**：自研预计 2 周开发 + 持续维护，LangGraph 1 天学习 +1 周开发。最终选择 LangGraph，将精力聚焦在业务逻辑而非基础设施。

---

#### Q2: 你的 Agent 系统里有 5 种角色，它们是如何协作的？

**考察点**：多 Agent 协作理解、系统架构

**回答**：
采用**状态机路由 + 并行执行 + 聚合响应**的架构：

```
用户输入 → Analyzer(意图识别) → 条件路由
                                   │
        ┌──────────────────────────┼───────────────────────┐
        ▼                          ▼                       ▼
    Plan 节点              Parallel Workers            Aggregator
  (计划制定)            (Tutor/Judge/Inquiry)         (响应融合)
                              │
                              ▼
                        Aggregator → 最终回复
```

**执行流程**：
1. **Analyzer 节点**：Pydantic 结构化输出识别 6 类意图（答疑/评审/探究/总结/计划/结束）
2. **条件路由**：根据意图动态路由到 Plan 节点或 Parallel Workers
3. **并行执行**：Tutor 和 Judge 可并发执行，Inquiry 依赖 Judge 输出串行执行
4. **Aggregator 融合**：将各模块输出融合成自然流畅的回复

**关键设计**：每个 Worker 有独立的缓存 Trace，支持缓存命中统计和性能优化。

#### Q3: Agent 里的记忆是怎么实现的？跨会话记忆怎么做？

**考察点**：记忆架构、长短期记忆管理

**回答**：
我们采用**三层记忆架构**：

| 层级 | 机制 | 作用 | 延迟 |
|------|------|------|------|
| **短期工作记忆** | 滑动窗口 12 条 | 保持对话流畅，解决指代消解 | O(1) |
| **长期认知链** | 摘要压缩（每 16 条触发） | 记录学习路径，防止遗忘 | LLM 调用 |
| **关联联想记忆** | Jaccard 召回 TopK=2 | 从全量历史精准召回 | 5ms |

**跨会话记忆实现**：
1. **会话存档**：每轮对话后持久化到 PostgreSQL（messages JSONB + conversation_summary）
2. **摘要压缩**：当未压缩消息≥16 条时，触发 LLM 蒸馏旧对话为"认知状态摘要"
3. **跨会话召回**：新会话开始时，加载历史摘要 + 使用 Jaccard 召回相关片段

**公式**：`上下文记忆 = 滑动窗口短期记忆 + 摘要压缩长期认知链 + 关联联想记忆`

#### Q4: 什么情况下不适合用摘要压缩上下文？

**考察点**：摘要压缩的局限性、技术边界

**回答**：
摘要压缩不适合以下场景：

1. **高信息密度对话**：每句话都包含关键信息（如法律条款讨论），压缩会丢失细节
2. **代码/公式密集场景**：代码片段、数学公式无法有效压缩，丢失语法结构
3. **低延迟要求场景**：LLM 压缩引入额外延迟（~500ms），实时对话体验受损
4. **成本敏感场景**：频繁调用 LLM 压缩，API 成本增加（约￥0.02/次）

**我们的应对策略**：
- 设置合理的压缩阈值（16 条），平衡成本和效果
- 保留滑动窗口（12 条），确保短期对话完整性
- 跨会话召回用 Jaccard（零成本），保证长尾相关性

#### Q5: 如果 Agent 进入死循环，你会怎么处理？如何预防？

**考察点**：异常处理、系统稳定性

**回答**：

**处理方案**（兜底）：
1. **超时熔断**：每个 Worker 设置 30s 超时，超时后返回空结果
2. **最大轮次限制**：单轮对话最多调用 3 次 LLM，超过直接返回兜底回复
3. **降级策略**：检测循环后切换到简单模式（仅 Tutor 答疑，关闭 Judge/Inquiry）

**预防方案**（提前设计）：
1. **状态追踪**：记录每轮对话的意图类型，检测连续 3 轮相同意图时触发熔断
2. **进度信号**：每个节点必须输出"进展标记"，无进展时终止循环
3. **显式退出条件**：在 System Prompt 中明确"如果用户连续 2 次表示不理解，直接给出答案"

**代码示例**：
```python
async def _run_worker_safe(worker_name, worker_coro, state):
    try:
        result = await asyncio.wait_for(worker_coro, timeout=30)
        return result if isinstance(result, dict) else {}
    except asyncio.TimeoutError:
        print(f"⚠️ Worker {worker_name} timeout")
        return {}
```

#### Q6: 你觉得什么情况下该用 Agent，什么情况下该用 RAG？

**考察点**：技术选型判断、场景理解

**回答**：

| 维度 | Agent | RAG |
|------|-------|-----|
| **核心能力** | 任务规划、工具调用、多轮交互 | 知识检索、信息增强 |
| **适用场景** | 需要多步决策、动态路由 | 需要外部知识增强 |
| **复杂度** | 高（状态管理、工具编排） | 低（检索 + 生成） |
| **延迟** | 高（多 LLM 调用） | 低（单次检索 + 生成） |

**明确选择 Agent 的场景**：
1. **多步骤任务**：如"帮我制定学习计划→执行→调整→总结"
2. **动态路由**：需要根据用户意图调用不同模块（答疑/评审/探究）
3. **工具依赖**：需要调用搜索、数据库、API 等外部工具

**明确选择 RAG 的场景**：
1. **知识库问答**：如"公司报销政策是什么"
2. **文档分析**：如"这份合同有哪些风险点"
3. **高频重复问题**：缓存命中率>80% 的场景

**我们的实践**：ChatTutor 是**Agent + RAG 混合架构**——Agent 负责任务路由，RAG 负责长尾知识召回。

#### Q7: 向量检索和混合检索分别适合什么场景？你们是分场景使用还是同时使用？

**考察点**：检索方案选型、实战经验

**回答**：

**向量检索（稠密）适用场景**：
- 语义相似度优先（如"机器学习"→"ML"）
- 跨语言检索
- 长尾、低频次查询

**混合检索（稠密 + 稀疏）适用场景**：
- 需要精确匹配（如产品型号、代码片段）
- 专业术语检索（如"O( n log n) 复杂度"）
- 高频、高价值查询（保证召回率）

**我们的实践**：
- **会话内召回**：使用 Jaccard（稀疏），延迟 5ms，零成本
- **跨会话召回**：使用向量检索（稠密），保证语义相关性
- **知识图谱检索**：混合检索（Jaccard + 共现分析）

**不是同时使用**，而是根据场景分层：会话内追求低延迟用 Jaccard，跨会话追求准确率用向量。

---

#### Q8: 稠密向量和稀疏向量的区别是什么？

**考察点**：向量检索基础

**回答**：

| 维度 | 稠密向量 (Dense) | 稀疏向量 (Sparse) |
|------|------------------|-------------------|
| **表示方式** | 低维稠密（768/1536 维） | 高维稀疏（词表维度，数万维） |
| **生成方式** | Embedding 模型（如 BGE、text2vec） | TF-IDF、BM25、SPLADE |
| **语义能力** | 强（"汽车"≈"车辆"） | 弱（"汽车"≠"车辆"） |
| **精确匹配** | 弱 | 强（专有名词、代码） |
| **延迟** | 中（需 Embedding 计算） | 低（倒排索引） |
| **存储** | 高（每文档 768 浮点数） | 低（倒排索引压缩） |

**公式对比**：
- 稠密：`向量相似度 = cos(emb_query, emb_doc)`
- 稀疏：`Jaccard = |A∩B| / |A∪B|`（词集合交集/并集）

#### Q9: RAG 里的召回阈值一般怎么设？你项目里的阈值是怎么定的？

**考察点**：阈值调优经验、数据驱动思维

**回答**：
召回阈值设定需要**平衡召回率和准确率**：

**通用方法**：
1. **标注测试集**：准备 100-200 条 query-doc 对，标注相关性
2. **绘制 PR 曲线**：计算不同阈值下的 Precision/Recall
3. **选择 F1 最高点**：或根据业务偏好（高召回/高准确）选择

**我们项目的阈值设定**：
- **Jaccard 相似度**：TopK=2（召回最相关的 2 条），不设固定阈值
- **向量检索**：余弦相似度≥0.75（通过测试集 F1 最高点确定）
- **混合检索**：RRF 融合后取 Top5

**阈值不合理的影响**：
- 阈值过高→漏召回（用户问 A，检索不到相关的 B）
- 阈值过低→噪声多（检索到不相关内容，干扰生成）

**动态调整策略**：如果用户反馈"回答不准确"，自动降低阈值重新检索。

#### Q10: 如果 RAG 系统上线后回答效果不好，你会怎么排查？

**考察点**：问题排查思路、系统性思维

**回答**：
采用**分层排查法**，从下到上：

```
检索层 → 生成层 → 反馈层
```

**Step 1: 检查检索层**
```bash
# 1. 召回测试
- 抽取 badcase 的 query，手动检索查看召回文档
- 检查：是否召回了相关文档？阈值是否过高？

# 2. 索引健康度
- 检查文档分片是否合理（是否切断了语义）
- 检查 Embedding 模型是否匹配（中文用 BGE-M3）
```

**Step 2: 检查生成层**
```bash
# 1. Prompt 分析
- 检查 System Prompt 是否清晰
- 检查是否注入了足够的上下文

# 2. 模型能力
- 换用更强模型（如 DeepSeek-V3 → GPT-4）测试
- 检查是否出现幻觉
```

**Step 3: 检查反馈层**
```bash
# 1. 用户反馈收集
- 点踩/点赞数据分析
- 低分回答聚类分析

# 2. A/B 测试
- 不同检索策略对比
- 不同 Prompt 版本对比
```

**优先级**：检索层 > 生成层 > 反馈层（80% 的问题出在检索）

#### Q11: 如果已经确认"能检索到正确文档"，但模型还是答错了，你会优先排查哪一层？

**考察点**：问题定位优先级、生成层理解

**回答**：
优先排查**生成层**，具体顺序：

**1. 上下文注入（最高优先级）**
- 检查：检索到的文档是否正确注入到 Prompt？
- 常见问题：检索结果被截断、关键信息丢失
- 验证：打印最终 Prompt，确认文档内容完整

**2. Prompt 设计**
- 检查：System Prompt 是否明确要求"基于上下文回答"
- 常见问题：模型忽略指令、自由发挥
- 优化：添加"如果上下文不足，请说明不确定"

**3. 证据绑定（Evidence Binding）**
- 检查：模型是否引用了正确的文档片段
- 优化：要求模型输出引用来源（如"[1]"）
- 示例：`"请基于以下文档回答，并在句末标注引用来源"`

**4. 模型能力**
- 检查：模型是否理解任务（复杂推理可能需要更强模型）
- 优化：换用 CoT Prompt 或更强模型

**实际案例**：我们的系统中出现过检索正确但回答错误，最终发现是 Prompt 中上下文被截断（超过 Token 限制），解决方式是增加摘要压缩。

#### Q12: 如果模型出现幻觉，你一般怎么处理？"证据绑定"具体是什么意思？

**考察点**：幻觉处理、事实一致性

**回答**：

**幻觉处理策略**：

| 策略 | 方法 | 效果 |
|------|------|------|
| **预防** | Prompt 要求"基于上下文回答" | 减少 50% 幻觉 |
| **检测** | 要求模型标注引用来源 | 可追溯 |
| **纠正** | 检索结果中添加"置信度"字段 | 低置信度时提示用户 |
| **兜底** | 添加"如果不确定，请说明" | 降低幻觉影响 |

**证据绑定（Evidence Binding）**：
指将生成的答案与检索到的文档片段建立明确的引用关系，让每个陈述都有据可查。

**实现方式**：
1. **引用标注**：要求模型在句末标注来源，如"随机森林是集成学习方法 [1]"
2. **片段 ID 追踪**：检索时给每个文档片段分配 ID，生成时绑定 ID
3. **反向验证**：生成后检查每个引用是否能在原文档中找到依据

**示例 Prompt**：
```
请基于以下文档片段回答用户问题。
要求：
1. 每个陈述必须在句末标注引用来源（如 [1]、[2]）
2. 如果文档中没有相关信息，请明确说明"根据现有资料无法回答"

文档片段：
[1] 随机森林是一种集成学习方法...
[2] 决策树是随机森林的基础组件...
```

#### Q13: 分片策略是怎么设计的？如果按字符或 Token 固定切分，把 Markdown 里的链接切断了怎么办？

**考察点**：文档分片策略、实战经验

**回答**：

**我们的分片策略**：
- **按语义分片**：优先按段落、章节切分，保持语义完整
- **重叠窗口**：相邻分片重叠 10-15%，防止上下文丢失
- **元数据保留**：每个分片保留文档标题、章节路径

**固定切分的问题**：
确实会遇到切断 Markdown 链接、代码块、表格的问题。

**解决方案**：
1. **Markdown 感知分片**：
   - 识别 `## `、`### ` 等标题标记，在标题处切分
   - 识别 ``` 代码块标记，保持代码块完整
   - 识别 `|` 表格标记，保持表格完整

2. **链接修复**：
   - 分片时检测 `[text](url` 是否被切断
   - 如果切断，向后扩展直到找到 `)` 闭合
   - 或在分片元数据中保留"父文档 ID"，生成时回溯

3. **后处理校验**：
   ```python
   def validate_chunk(chunk):
       # 检查 Markdown 语法完整性
       if chunk.count('```') % 2 != 0:  # 代码块未闭合
           return False
       if chunk.count('[') != chunk.count(']'):  # 链接未闭合
           return False
       return True
   ```

#### Q14: 如果遇到高频重复问题，你会怎么优化处理？

**考察点**：性能优化、缓存策略

**回答**：
采用**多层缓存 + 语义去重**策略：

**1. 语义缓存（Semantic Cache）**
```python
# 基于 Jaccard 相似度检测重复问题
def is_duplicate(query, cache, threshold=0.85):
    for cached_query in cache:
        if jaccard_similarity(query, cached_query) >= threshold:
            return cache[cached_query]  # 命中缓存
    return None
```

**2. 双层 TTL 缓存**
- **L1 缓存**：内存缓存（Redis），TTL=5 分钟，存储最近 100 个问答对
- **L2 缓存**：数据库缓存（PostgreSQL），TTL=24 小时，存储高频问答对

**3. 效果**
| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 缓存命中率 | 0% | 45% |
| API 成本 | ￥0.05/轮 | ￥0.02/轮 |
| 响应延迟 | 3.2s | 1.6s |

**4. 缓存失效策略**
- 检测到用户提到"不对"、"重新回答"时，自动清除缓存
- 定期清理低命中率缓存（LRU 策略）

#### Q16: 如果 Prompt 的回答需要依赖"当前时间"和"当前用户身份"等外部信息，你会怎么设计？

**考察点**：工程与 Prompt 的边界、信息注入

**回答**：
**不会写在 Prompt 里**，而是通过**代码/工具注入**：

**方案 1：System Prompt 动态拼接**
```python
def build_prompt(user_id, current_time):
    user_info = get_user_profile(user_id)  # 从数据库获取
    prompt = f"""当前时间：{current_time}
用户信息：{user_info['name']}, {user_info['level']}
...（其余 Prompt）"""
    return prompt
```

**方案 2：工具调用（推荐）**
```python
# 定义工具
@tool
def get_current_time() -> str:
    return datetime.now().strftime("%Y-%m-%d %H:%M")

@tool
def get_user_info(user_id: str) -> dict:
    return db.query(User).filter_by(id=user_id).first()

# 模型按需调用
model_with_tools = model.bind_tools([get_current_time, get_user_info])
```

**方案 3：中间件注入（分步骤获取）**
```python
# 在请求中间件中获取
@app.middleware("http")
async def add_context(request, call_next):
    request.state.user = await get_current_user(request)
    request.state.time = datetime.now()
    response = await call_next(request)
    return response
```

**选择原则**：
- 静态信息（如角色定义）→ 写在 Prompt
- 动态信息（如时间、用户）→ 代码注入
- 按需信息（如用户历史）→ 工具调用

#### Q17: 你提到做过文本清洗，这部分具体做了什么？

**考察点**：数据预处理经验

**回答**：

**文本清洗内容**：

1. **特殊字符处理**
   - 移除零宽字符（`\u200b`、`\u200c`）
   - 移除控制字符（`\x00-\x1f`）
   - 标准化空白字符（多个空格→单个空格）

2. **Markdown 修复**
   - 修复被切断的链接（检测 `[` 无 `]` 配对）
   - 修复未闭合的代码块（检测 ` ``` ` 奇数个）
   - 标准化标题格式（`# 标题`→`## 标题`）

3. **公式标准化**
   - 行内公式：`$...$` 确保无换行
   - 独立公式：`$$...$$` 确保前后有换行
   - 转义处理：`\$` → `$`

4. **HTML 实体转义**
   - `&nbsp;` → 空格
   - `&lt;` → `<`
   - `&gt;` → `>`

**代码示例**：
```python
def clean_text(text):
    # 移除零宽字符
    text = re.sub(r'[\u200b-\u200d]', '', text)
    # 标准化空白
    text = re.sub(r'\s+', ' ', text).strip()
    # 修复 Markdown 链接
    text = fix_markdown_links(text)
    return text
```

#### Q18: 你的问答系统是不是所有场景都用向量型 RAG 来解决？

**考察点**：技术方案多样性、场景判断

**回答**：
**不是**，我们采用**分层检索策略**：

| 场景 | 检索方式 | 理由 |
|------|----------|------|
| **会话内召回** | Jaccard 相似度 | 字面匹配足够，延迟 5ms，零成本 |
| **跨会话召回** | 向量检索 | 需要语义理解，保证长尾相关性 |
| **知识图谱检索** | 混合（Jaccard + 共现） | 实体匹配需要精确，关系需要语义 |
| **高频重复问题** | 语义缓存 | 直接返回缓存答案，无需检索 |
| **实时信息问答** | 搜索工具 | RAG 无法覆盖最新信息 |

**选型原则**：
- 追求低延迟→Jaccard
- 追求语义准确→向量
- 追求成本效益→混合
- 追求实时性→搜索工具

#### Q19: 除了摘要之外，还有哪些常见的上下文优化手段？

**考察点**：上下文管理技术广度

**回答**：

| 手段 | 原理 | 适用场景 |
|------|------|----------|
| **滑动窗口** | 保留最近 N 条 | 短期对话流畅 |
| **摘要压缩** | LLM 蒸馏历史 | 长对话防遗忘 |
| **向量检索召回** | Embedding + 相似度 | 跨会话关联 |
| **关键词召回** | Jaccard/TF-IDF | 会话内快速检索 |
| **分层存储** | 重要信息单独存储 | 用户画像、学习计划 |
| **增量更新** | 只追加不覆盖 | 低延迟场景 |
| **Token 预算分配** | 动态分配上下文 Token | 成本敏感场景 |

**我们的实践**：
- 三层记忆架构 = 滑动窗口 + 摘要压缩 + Jaccard 召回
- 用户画像单独存储（learner_profiles 表），每轮对话前注入
