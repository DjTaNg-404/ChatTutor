# ChatTutor 知识图谱 Agent 项目 - 面试逐字稿（2000 字）

---

各位面试官好，我今天要分享的项目是 **ChatTutor 知识图谱 Agent 系统**。这是一款智能学习伴侣产品，核心理念是"一场对话，就是一次学习"。

## 一、项目背景

在做这个项目之前，我们观察到市面上 AI 教育产品有三大痛点：

**第一，对话无记忆。** 大多数产品超过 10 轮对话就开始遗忘，用户需要反复解释之前讨论过的内容。

**第二，教学无个性。** 无法根据学生的学习进度和偏好调整策略，永远是"千人一面"的回答。

**第三，知识无沉淀。** 学习内容散落在对话记录里，无法形成结构化知识体系，学完就忘。

针对这些问题，我设计并开发了 ChatTutor 系统，目标是实现：100 轮以上连续对话不遗忘、6 种意图精准识别、知识图谱自动构建、响应时间小于 2 秒。

## 二、核心架构

整个系统分为三大模块：**多 Agent 协作引擎**、**三层记忆架构**、**知识图谱系统**。

### 2.1 多 Agent 协作引擎

我选用了 **LangGraph** 作为状态机框架。为什么选它而不是 LangChain？因为 LangChain 是有向无环图（DAG），不支持循环；而 LangGraph 支持循环图和多 Agent 协作，更适合复杂对话场景。

系统包含 5 个 Agent 角色：

- **Analyzer（大脑）**：负责意图识别和路由决策
- **Tutor（答疑者）**：解答用户疑问，支持搜索增强
- **Judge（评审员）**：评估用户回答的逻辑漏洞
- **Inquiry（探究者）**：提出启发式问题引导思考
- **Aggregator（汇总者）**：融合各 Agent 输出并存档

状态机流程是这样的：用户输入 → Analyzer 识别意图 → 条件路由到对应 Worker 节点 → 并行执行 → Aggregator 汇总输出。

### 2.2 意图识别设计

这里我用到了 **Pydantic 结构化输出**。定义一个 ExecutionPlan 模型，包含 6 个布尔字段：

- needs_tutor_answer：是否需要答疑
- needs_judge：是否需要评估
- needs_inquiry：是否需要追问
- request_summary：是否请求总结
- request_plan：是否请求计划
- is_concluding：是否想结束对话

还有一个 thought_process 字段，强制模型先输出推理过程再给结果。这是 **思维链（Chain of Thought）** 技术的应用，研究表明能提升 15% 以上的准确率。

实际效果：规则匹配准确率 65%，文本分类 82%，结构化输出达到 **92.3%**。

### 2.3 三层记忆架构

这是解决长对话遗忘的核心设计。

**第一层：短期工作记忆**。保留最近 12 条原始对话，解决指代消解问题，比如"它"指什么。

**第二层：长期认知链**。每 16 条对话触发一次 LLM 蒸馏压缩，保留最后 5 条原始消息避免信息损失。这里用的是增量压缩策略，只处理新增消息，不重复处理已压缩部分。

**第三层：关联联想记忆**。基于 Jaccard 相似度从全量历史中召回 Top 2 相关片段。为什么用 Jaccard 不用向量检索？因为会话内召回对语义要求不高，Jaccard 零成本、5ms 延迟，而向量检索需要 150ms 加 Embedding 服务成本。

三层配合下来，系统支持 **150 轮以上对话不遗忘**。

### 2.4 知识图谱系统

知识图谱的构建分三步：实体识别、关系抽取、图谱优化。

**实体识别**采用双模式混合架构：
- NER 模式：用 ckiplab/bert-base-chinese-ner 处理通用实体，速度快（<50ms/条）
- LLM 模式：用 DeepSeek 处理领域术语，准确率高

分流策略是：先跑 NER，置信度低于 0.5 的再用 LLM 补全。80% 的实体用 NER 处理，20% 用 LLM，总成本降低 60%，准确率仅损失 2%。

**关系抽取**也有两种模式：
- 共现分析：基于距离加权计算关系强度，适合高频简单关系
- LLM 抽取：处理语义复杂的低频关系

融合公式是：最终强度 = 0.6×距离强度 + 0.4×实体置信度

**图谱优化**有 4 种策略：
1. 语义归一化（阈值 0.8）：把"机器学习"和"ML"合并
2. 传递性约简（阈值 0.7）：A→B→C 存在时删除 A→C
3. 属性图转换：将属性从节点改为边
4. 统计过滤（阈值 0.3）：删除低频噪音节点

最终实体识别准确率达到 **87.1%**。

## 三、模型选型

为什么选 DeepSeek 而不是 GPT-4？

从价格看：DeepSeek 输入$0.27/1M token，GPT-4 是$10，相差 37 倍。

从中文能力看：DeepSeek 是国产模型，中文语义理解明显优于 GPT-4。

从延迟看：DeepSeek 平均 500ms，GPT-4 是 1000ms。

具体配置是：
- 实体提取、关系推断用 deepseek-chat，温度 0.1 保证稳定性
- 图谱总结用 deepseek-chat，温度 0.3 允许一定创造性
- 对话生成温度 0.7 增加多样性

## 五、经验沉淀

这个项目有三个核心亮点：

**第一，模块化分层。** 知识图谱、Agent、记忆系统完全解耦，可以独立迭代和测试。代码复用率达到 75%。

**第二，混合方案。** NER+LLM、向量+Jaccard 的混合策略平衡了准确率和成本。这是工程实践中非常重要的"性价比"思维。

**第三，状态机架构。** LangGraph 的条件路由让复杂对话流变得可控，比手写状态机可维护性高很多。

当然也有可以改进的地方：
- 缺少监控体系，后续会增加 Prometheus+Grafana
- Jaccard 可以升级到 BGE-M3 提升语义召回
- 没有 A/B 测试框架，优化靠经验而非数据

## 六、面试问答

### Q1: 为什么选 LangGraph 而不是自研状态机？

A: 我们评估过自研，但 LangGraph 有四个优势：与 LangChain 生态无缝集成、Graph 定义可可视化、支持循环和并行高级模式、社区活跃问题容易解决。成本对比是自研要 2 周开发加持续维护，LangGraph 只要 1 天学习加 1 周开发。

### Q2: 三层记忆为什么用 Jaccard 不用向量？

A: 这是成本和收益的权衡。Jaccard 准确率 85%、延迟 5ms、零成本；向量检索准确率 92%、延迟 150ms、每次调用要钱。会话内召回对语义要求不高，字面匹配足够，所以用 Jaccard。跨会话召回才用向量保证长尾相关性。

### Q3: 如果重新设计会做什么改动？

A: 优先级排序是：第一增加监控体系，第二支持流式响应降低感知延迟，第三升级到 BGE-M3 提升召回，第四建立 A/B 测试框架。监控是第一位的，因为上线后看不到 API 延迟和错误率，优化没有数据支撑。
